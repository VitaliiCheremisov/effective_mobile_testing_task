Этап 1. Проектирование БД (уточнённое под ТЗ)

Цель — спроектировать простую, но расширяемую RBAC‑схему, строго соответствующую формулировкам задания.

Основные таблицы:

1. `users`
   - `id` (PK)
   - `full_name`
   - `email` (уникальный)
   - `password_hash`
   - `is_active` (bool, по умолчанию `True`; soft delete = `False` по требованию ТЗ)
   - `created_at`, `updated_at`

2. `roles`
   - `id` (PK)
   - `name` (уникальный, например `admin`, `manager`, `user`)
   - `description`

3. `permissions`
   - `id` (PK)
   - `codename` (уникальный, например `can_view_reports`, `can_manage_roles`)
   - `name` (человекочитаемое название)

4. `role_permissions`
   - `id` (PK)
   - `role_id` (FK → `roles`)
   - `permission_id` (FK → `permissions`)
   - уникальный индекс по паре (`role_id`, `permission_id`)

5. `user_roles`
   - `id` (PK)
   - `user_id` (FK → `users`)
   - `role_id` (FK → `roles`)
   - уникальный индекс по паре (`user_id`, `role_id`)

6. `token_blacklist`
   - `id` (PK)
   - `token_jti` (или хеш токена)
   - `expires_at` (дата/время истечения токена)

Особенности:
- **Soft delete**: пользователь «удаляется» переводом `is_active=False`, запись в БД сохраняется (как требуется в ТЗ).
- **Logout через blacklist**: при выходе в `token_blacklist` заносится `jti`/хеш токена до его срока истечения.
- **Расширяемость**: новые роли и права можно добавлять через БД и админ‑API без изменения кода.
- В `README.md` планируется ER‑диаграмма (Mermaid), описывающая эту схему.


Этап 2. Стек и базовая настройка

- **Framework**: Django + DRF.
  - Не использовать `django.contrib.auth.models.User` и `AbstractUser`.
  - Не использовать `rest_framework.authtoken` и встроенную токен‑авторизацию.
  - Разрешён только модуль хеширования паролей: `django.contrib.auth.hashers`.
- **База данных**: PostgreSQL.
- **Контейнеризация**: `docker-compose` с сервисами `web` (Django) и `db` (Postgres).
- **Аутентификация**: собственная реализация JWT на базе `PyJWT`:
  - payload: `user_id`, `exp`, `jti` (уникальный идентификатор токена);
  - отдельный `JWT_SECRET_KEY` в переменных окружения.

В `settings.py` явно указать свои классы аутентификации и авторизации:

```python
REST_FRAMEWORK = {
    "DEFAULT_AUTHENTICATION_CLASSES": [
        "apps.auth_core.backends.JWTAuthentication",
    ],
    "DEFAULT_PERMISSION_CLASSES": [
        "apps.rbac.permissions.RBACPermission",
    ],
}
```

Проектная структура (план):

- `apps/users/` — модель пользователя, регистрация, профиль.
- `apps/auth_core/` — JWT, blacklist, кастомный DRF `BaseAuthentication`.
- `apps/rbac/` — модели ролей и прав, DRF `BasePermission`, сервис проверки прав.
- `apps/business/` — mock‑вьюхи для демонстрации системы доступа.


Этап 3. Ядро аутентификации (AuthN)

1. **Модель `User` (`apps/users/models.py`)**
   - Обычная Django‑модель (`models.Model`), без наследования от `AbstractUser`.
   - Методы:
     - `set_password(raw_password)` — обёртка над `make_password`.
     - `check_password(raw_password)` — обёртка над `check_password`.
   - Поле `is_active`:
     - логическое, управляет доступом к системе;
     - при soft delete устанавливается в `False`.

2. **Менеджер `User`**
   - Базовый `QuerySet` по умолчанию возвращает только `is_active=True`.
   - Для операций вроде логина использовать отдельный менеджер/метод, который умеет явно проверять `is_active` и при необходимости работать с неактивными пользователями (для диагностики).

3. **JWT‑логика (`apps/auth_core`)**
   - Генерация access‑токена:
     - минимум поля: `user_id`, `exp`, `jti`.
   - Для уровня Junior достаточно access‑токена; при наличии времени можно добавить refresh‑токен (плюс, но не обязательное требование).

4. **Logout и blacklist**
   - Эндпоинт logout принимает текущий JWT из заголовка.
   - Токен декодируется, из payload берётся `jti` и `exp`.
   - В таблицу `token_blacklist` добавляется запись (`token_jti`, `expires_at`).
   - В кастомном `JWTAuthentication` при каждом запросе:
     - декодировать токен;
     - проверить, что `jti` не находится в `token_blacklist` (и не истёк).
   - Таким образом, после logout токен становится фактически недействительным до своего `exp`.

5. **Чёткое разделение AuthN/AuthZ**
   - AuthN (кто ты?) — полностью в классе `JWTAuthentication`:
     - отсутствие/битый токен → **401 Unauthorized**;
     - неактивный пользователь (`is_active=False`) → **401 Unauthorized**.
   - AuthZ (что тебе можно?) — полностью в permission‑классах RBAC:
     - пользователю чего‑то нельзя по ролям/правам → **403 Forbidden**.


Этап 4. Авторизация (AuthZ) и админ‑API

1. **RBAC‑модели (`apps/rbac/models.py`)**
   - Реализация таблиц `roles`, `permissions`, `role_permissions`, `user_roles`.
   - Связи многие‑ко‑многим через явные промежуточные таблицы с уникальными индексами.

2. **Сервис проверки прав (`apps/rbac/service.py`)**
   - Функция/класс `has_permission(user, permission_codename)`:
     - получить все роли пользователя;
     - по ролям получить все связанные permissions;
     - проверить наличие `codename`.
   - Возможное кэширование прав пользователя в рамках одного запроса.

3. **Permission‑класс DRF (`apps/rbac/permissions.py`)**
   - Базовый класс `RBACPermission(BasePermission)` с атрибутом `required_permission`.
   - В `has_permission`:
     - если `request.user` не аутентифицирован → доверяем `JWTAuthentication` (он вернёт 401);
     - если пользователь аутентифицирован, но `has_permission(user, required_permission)` вернул `False` → **403 Forbidden**.
   - Конкретные вьюхи задают `required_permission`, например `"can_view_reports"`.

4. **Админ‑API управления ролями и правами**

Namespace (условно): `/api/admin/` (отличать от бизнес‑эндпоинтов).

Примеры эндпоинтов:

- `POST /api/admin/roles/` — создать роль.
- `GET /api/admin/roles/` — список ролей.
- `POST /api/admin/roles/{id}/permissions/` — добавить право роли.
- `DELETE /api/admin/roles/{id}/permissions/{perm_id}/` — убрать право.
- `POST /api/admin/users/{id}/roles/` — назначить роль пользователю.
- `DELETE /api/admin/users/{id}/roles/{role_id}/` — снять роль с пользователя.

Защита:
- эти эндпоинты доступны только пользователю:
  - с ролью `admin`;
  - и с отдельным правом, например `can_manage_roles`.
- это явно демонстрирует гибкость системы и соответствие пункту ТЗ про «API для изменения правил».

Этап 3. Базовый функционал пользователей и безопасность

1. **Модель пользователя**
   - Собственная модель `User`, наследующаяся от `models.Model` (без `AbstractUser`).
   - Хеширование пароля через `django.contrib.auth.hashers.make_password` / `check_password` (используется только механизм хеширования, без встроенных моделей и view).

2. **Регистрация**
   - Эндпоинт `/api/v1/auth/register/`.
   - Входные данные: ФИО, email, пароль, подтверждение пароля.
   - Проверки: уникальность email, совпадение паролей, базовая валидация сложности пароля.

3. **Login / Logout**
   - Эндпоинт логина `/api/v1/auth/login/`: по email и паролю.
   - Проверка:
     - что пользователь не помечен как `is_deleted`;
     - что пароль корректен.
   - При успешном логине — генерация JWT и возврат в ответе.
   - Эндпоинт logout `/api/v1/auth/logout/`:
     - получение токена из заголовка;
     - помещение токена (или его `jti`) в blacklist до окончания его жизни.

4. **Обновление и удаление пользователя**
   - Эндпоинт `/api/v1/users/me/`:
     - `GET` — получение своих данных;
     - `PATCH` — изменение ФИО, email (с проверкой уникальности).
   - Эндпоинт мягкого удаления `/api/v1/users/me/delete/`:
     - установка `is_deleted = True`;
     - инвалидирование текущего токена (logout);
     - дальнейший login невозможен.


Этап 4. Собственная система аутентификации и авторизации

1. **Custom authentication backend для DRF**
   - Класс, наследующийся от `BaseAuthentication`.
   - Логика:
     - извлечь токен из заголовка `Authorization: Bearer <token>`;
     - декодировать JWT (проверка подписи и срока годности);
     - по `user_id` из payload получить пользователя;
     - проверить, что пользователь существует и не удалён;
     - проверить, что токен не в blacklist.
   - При неуспехе — выбрасывать `AuthenticationFailed` → клиент получает **401 Unauthorized**.

2. **Модель и слой авторизации (RBAC)**
   - Сопоставление: пользователь → роли → разрешения.
   - Отдельный сервис/утилита, например `rbac_service.has_permission(user, permission_code)`, инкапсулирующий доступ к БД и кэширование при необходимости.

3. **Custom permission classes для DRF**
   - Класс, наследующийся от `BasePermission`, с параметром `required_permission`.
   - В `has_permission`:
     - если пользователь не аутентифицирован — 401 (делает authentication backend);
     - если аутентифицирован, но нет нужного разрешения — вернуть False → **403 Forbidden**.
   - Вьюхи используют разрешения так, чтобы вся проверка прав была вынесена из бизнес‑логики.

4. **Описание модели доступа**
   - В `README.md` (или отдельном `docs/access_model.md`) текстово описать:
     - сущности `Role` и `Permission`;
     - пример матрицы ролей и прав (кто что может делать);
     - как администратор может изменять правила через API.
